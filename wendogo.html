My contributions.

// Networking architecture — server class foundation

I designed and built the entire networking architecture from scratch using Unity Netcode for GameObjects. The core of my work was a dedicated server-side class centralizing all server methods — a clean API that my two co-programmers could extend as they added new features.

When a player performs an action, it's sent to the server as a command. The server validates it against the current game state, applies the effect if valid, then broadcasts the updated state to all clients. Invalid actions are rejected and only the sender is notified. This keeps state consistent across all clients regardless of network conditions.

By exposing a clear server API — adding methods to the server class as needed — the team could build new game features without touching the network layer. This separation made collaboration smoother across 3 programmers.

// Card mechanics hierarchy

I built the card system architecture — the class hierarchy that defines how cards work. Each card type inherits from an abstract CardEffect base class and overrides the effect logic. Adding a new card type means: create a class, override one method.

Card data is stored in a CardDatabaseSO ScriptableObject backing a DeckConfiguration — easy to configure in the editor without touching code.

// Turn order and day/night loop

I implemented the sequential turn system (P1 → P2 → P3 → P4) for both phases, along with the dawn reveal — aggregating all hidden night actions and surfacing their consequences at the start of the next day.
